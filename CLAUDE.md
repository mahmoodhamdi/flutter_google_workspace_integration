## Architecture

Clean architecture per feature. Domain layer is **pure Dart** ŌĆö no Flutter
imports. Data layer implements domain interfaces. Presentation layer uses
Riverpod and depends only on domain.

```
lib/
  core/
    auth/       # AuthRepository + secure storage + token refresh
    errors/     # AppError sealed class, guard helpers
    network/    # Dio with auth interceptor
    storage/    # Hive bootstrap + sync queue
    routing/    # GoRouter with auth-aware redirects
    theme/      # Per-flavor Material 3
    config/     # AppConfig + AppFlavor + feature flags
    notifications/
  features/
    google_calendar/ google_drive/ google_sheets/ google_gmail/
    google_contacts/ google_maps/  google_meet/   home/  settings/
  l10n/         # ARB files (EN + AR)
  main*.dart    # Per-flavor entry points
```

## State management

**Riverpod 2.x only.** Do not introduce `flutter_bloc`, `get_it`, or any other
DI/state library. Use `AsyncNotifier` / `AsyncNotifierProvider.family` for
async screen state. Use `Provider` for repositories and services.

## Error handling

Every repository method returns `Future<Result<T>>` (which is
`Future<Either<AppError, T>>`). Throw inside `guard()` or `guardWithRetry()`,
never let raw exceptions escape a repository.

`AppError` is a sealed freezed class ŌĆö when matching, use `switch` exhaustively;
don't `is`-check.

## OAuth scope discipline (CRITICAL)

**Never add a RESTRICTED scope without explicit approval and a documented
business case in `.agent/decisions.md` or `sales/OAUTH_SCOPES_EXPLAINED.md`.**

Restricted scopes trigger Google's CASA Security Assessment ($15K-75K +
3-6 months wait), which destroys the product's marketability. The whole
architecture is designed around SENSITIVE scopes only.

Acceptable scopes are listed in `lib/core/utils/constants/api_constants.dart`.
Adding a new feature? Pick the most-restrictive scope that achieves the use case.

## Codegen

Run `dart run build_runner build --delete-conflicting-outputs` after editing
any `*.dart` file with `@freezed`, `@JsonSerializable`, or `@riverpod`
annotations. Generated files (`*.g.dart`, `*.freezed.dart`) are gitignored.

## Tests

- Unit tests live alongside the layer they test under `test/`
- Widget tests in `test/widget/`
- Golden tests in `test/golden/`
- New features ship with tests; PRs without test coverage will be rejected
- Mock services with `mocktail`; never call real network from a unit test

## Commits

Conventional commits: `feat(scope): ...`, `fix(scope): ...`, `chore: ...`,
`docs: ...`, `test: ...`, `ci: ...`. Scope = feature name (`calendar`, `drive`,
etc.) or layer (`core`, `auth`, `network`).

**Never include AI attribution in commit messages** (no "Co-Authored-By:
Claude", no `­¤ż¢ Generated with...`, no "Generated by"). This is a production
codebase shipping commercially.

## Verticals

Each vertical (`bizcalendar`, `drivevault`, `sheetsops`, `meetcompanion`) is a
**Flutter flavor**, not a separate app. The `FLAVOR` `--dart-define` selects
the `AppConfig` which decides feature flags, branding, and OAuth scopes.

Build: `flutter build apk -t lib/main_bizcalendar.dart --dart-define=FLAVOR=bizcalendar`

## Things we explicitly do NOT do

- Do not implement Google Docs editing (see `.agent/decisions.md` D1)
- Do not implement Google Keep (no official API ŌĆö see D2)
- Do not implement Google Analytics on mobile (see D3)
- Do not use `flutter_bloc` (see D6 ŌĆö using Riverpod)
- Do not use `get_it` (see D6 ŌĆö using Riverpod providers as DI)
- Do not use `gmail.modify` or `gmail.readonly` (RESTRICTED ŌĆö see D4)
- Do not use full `drive` scope (RESTRICTED ŌĆö see D5)
